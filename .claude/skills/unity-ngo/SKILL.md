## Unity NGO (Netcode for GameObjects) 아키텍처 및 동기화 가이드라인

이 프로젝트의 네트워크 코드는 철저하게 **서버 권한(Server-Authoritative) 모델**을 따릅니다. 코드를 작성할 때 다음의 네트워크 설계 원칙을 기계적으로 준수하십시오.

### 1. 서버 권한 (Server-Authoritative) 원칙
- **서버는 게임 상태의 유일한 진실의 원천(Source of Truth)입니다**.
- 클라이언트는 오직 액션(행동)을 요청할 뿐이며, 서버가 이를 검증(Validate)하고 결과를 적용해야 합니다.
- **서버 소유 상태의 예시:** 플레이어 HP, 전투 데미지 판정, 객체 스폰/디스폰, 인벤토리 상태, 권한이 필요한 위치/상태 데이터 등.

### 2. NetworkVariable vs RPC의 명확한 분리
네트워크 데이터를 동기화할 때, 이것이 '지속되는 상태'인지 '일회성 사건'인지 명확히 구분하십시오.
- **NetworkVariable:** 체력, 현재 장착 무기 등 **지속적으로 동기화되어야 하는 상태(Persistent synchronized state)**에만 사용하십시오. 불필요하게 쓰기 빈도를 높여(Spam) 네트워크 대역폭을 낭비하지 마십시오.
- **RPC (Remote Procedure Call):** 데미지 발생 알림, 파티클 재생 요청 등 **일회성 이벤트나 요청(One-time events or requests)**에만 사용하십시오. RPC를 상태 동기화 목적으로 남용해서는 안 됩니다.
- 클라이언트 측의 단순한 시각적(Visual) 상태가 서버의 게임플레이 권한 상태로 둔갑하지 않도록 주의하십시오.

### 3. 클라이언트 측 예측 (Client-Side Prediction)
응답성을 높이기 위해 클라이언트 측 예측 로직을 도입할 경우, 다음을 준수하십시오:
- 예측 로직이 들어간 코드 경로(Path)는 주석이나 메서드명으로 명확히 라벨링하십시오.
- 클라이언트가 예측하여 보여주는 시각적 요소와, 서버가 판정하는 실제 권한 처리 로직을 코드 레벨에서 완벽히 분리하십시오.
- 예측 실패 시 상태를 복구(Reconciliation)하는 가정을 명확히 문서화하십시오.
- 예측된 상태를 서버가 확정한 최종 진실(Final server truth)과 조용히 뒤섞지 마십시오.

### 4. 대역폭 최적화 (Bandwidth Awareness)
네트워크 통신은 비용이 매우 큽니다.
- `NetworkVariable` 등의 동기화 데이터는 가능한 한 쓰기 빈도를 줄이십시오.
- 값이 변경되지 않았음에도 불필요하게 동일한 값을 서버나 클라이언트로 전송하지 마십시오.
- 로컬 클라이언트에게만 의미 있는 순수 코스메틱(시각적 꾸밈) 상태는 네트워크를 통해 동기화하지 마십시오.
- 지속적인 상태 갱신이 필요 없는 데이터라면, 합리적인 선에서 이벤트 기반(Event-driven) 동기화를 우선적으로 고려하십시오.
